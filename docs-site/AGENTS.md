# Scion Documentation Meta-Guide

This guide outlines the organizational philosophy, architecture, and curation standards for the Scion documentation project. It serves as a blueprint for both human and AI contributors to ensure a consistent and navigable documentation experience.

## 1. The Primary Axis: Operational Mode

Scion documentation is organized on the **operational-mode spine** defined in the
root [`GLOSSARY.md`](../GLOSSARY.md) — the single source of truth for terminology.
The site is **not** organized by persona. Instead, readers self-select by the
deployment target they are running:

> **Local → Workstation → Single-node hosted → HA hosted**

This spine is a ladder of increasing infrastructure, separated by two orthogonal
dimensions:

| Dimension | Meaning | Control |
| :--- | :--- | :--- |
| **Availability tier** | Whether the Hub is a single instance on an embedded database or replicated on an external one. | `SCION_SERVER_DATABASE_DRIVER` (`sqlite` vs `postgres`) |
| **Tenancy** | Single-user vs multi-user. Only opens up once hosted. | Orthogonal to the tier |

| Mode | Control plane | State & durability | Tenancy |
| :--- | :--- | :--- | :--- |
| **Local** | None (CLI only) | Local machine; git-worktree isolation | Single-user |
| **Workstation** | Combo server (Hub + Runtime Broker + Web) on loopback | Embedded SQLite | Single-user |
| **Single-node hosted** | One networked Hub on a single node | Embedded SQLite; non-HA | Single- or multi-user |
| **HA hosted** | Hub replicated behind a load balancer | External Postgres + object storage; HA | Single- or multi-user |

### The secondary fork: user vs admin

Inside each mode, content forks into a **user** and an **admin** track. The depth of
the fork scales with the mode's complexity:

- **Local** — single audience, no admin.
- **Workstation** — single audience; "admin" tasks (start the combo server, pick a
  runtime, network bridge) are light and self-served.
- **Single-node hosted** — moderate user/admin split.
- **HA hosted** — the fullest user/admin split (Postgres, object storage,
  load-balanced replicas, brokers, IAP, RBAC, observability).

The hosted **user** journey is shared by both hosted tiers. Its pages live once
under `hosted/user/` and are referenced from both tiers' sidebars — link, don't
duplicate.

## 2. Content Architecture

The sidebar (`astro.config.mjs`) reflects the mode axis directly:

1. **Introduction & Foundations** — *Everyone, mode-agnostic.* Overview, Choosing a
   Mode, Core Concepts, Philosophy, Supported Harnesses, Glossary, Release Notes.
2. **Getting Started (Workstation)** — Install, the Onboarding Wizard, first-agent
   Tutorial.
3. **Local Mode** — *Single audience.* CLI-only usage: agent lifecycle, workspaces &
   sharing modes, templates, credentials, custom images, tmux, completions, local
   configuration.
4. **Workstation Mode** — *Single audience; the hosted experience locally.* The combo
   server, the Web Dashboard, Hub-managed vs Linked projects.
5. **Hosted — Single-node** — *Cheap networked Hub.* User Guide + Admin Guide (one
   VM / Cloud Run + SQLite, auth & tenancy, secrets, observability).
6. **Hosted — HA** — *Durable, always-on.* User Guide (shared) + Admin Guide
   (Postgres + object storage + LB replicas, brokers & profiles, Kubernetes,
   identity & access, lifecycle hooks, observability).
7. **Technical Reference** — *Mode-agnostic; autogenerated from `reference/`.* CLI,
   API, config schemas, security.
8. **Contributing** — *Autogenerated from `contributing/`.* Architecture, ADK,
   harness-dev, runtime-dev, design catalog, logging. (The former "Development"
   section is folded in here.)

### Directory layout

Content directories mirror the mode axis:

```
src/content/docs/
├── (foundations pages at root: overview, choosing-a-mode, concepts, …)
├── getting-started/     (install, onboarding, tutorial)
├── local/               (Local Mode)
├── workstation/         (Workstation Mode)
├── hosted/
│   ├── user/            (shared hosted user journey)
│   ├── single-node/     (Single-node admin)
│   └── ha/              (HA admin)
├── reference/           (autogenerated Technical Reference)
└── contributing/        (autogenerated Contributing)
```

## 3. Vocabulary

Use current terminology from the root `GLOSSARY.md`. **Do not** use deprecated terms:

| Deprecated | Use instead |
| :--- | :--- |
| Grove | Project |
| Solo mode | Local mode |
| "Mode 3" (numeric) | HA hosted |
| bare "Broker" | Runtime Broker (a *service*), or a specific broker type (Node-Bound / Proxy / Embedded / Hosted) |
| "two workspace strategies" | the three sharing modes: Shared-plain, Worktree-per-agent, Clone-per-agent |

Key distinctions to preserve:

- A **Runtime Broker** is a *service*, not a machine. A machine that offers runtimes
  is a *node-bound broker*.
- A **Project** is not a **Group**. A Group is a permissions-system collection of
  users (multi-user, hosted only).
- **Harnesses are not plugins.** They are external, vendor-supplied agent software
  Scion drives, integrated via a harness-config, not the `go-plugin` interface.
- **Message Broker** (chat/notification transport) ≠ **Event Bus** (internal event
  fan-out). Keep them distinct.

## 4. Config Files

Scion accepts **both** `settings.yaml` and `settings.json` on disk, in precedence
order `settings.yaml` → `settings.yml` → `settings.json` (YAML preferred). `scion init`
writes YAML by default. The JSON schema lives at
`pkg/config/schemas/settings-v1.schema.json`. Never show a JSON object inside a
` ```yaml ` fence.

## 5. Curation Standards

1. **Code-First Truth**: Documentation should be updated in the same PR as the feature
   implementation. Docs follow code — if they diverge, fix the docs and flag it.
2. **Source of Truth**:
   - `GLOSSARY.md` (root) for terminology and the mode spine.
   - `.design/` for architectural intent.
   - `pkg/` for implementation details; verify CLI flags in `cmd/`.
3. **Mode-Specific Placement**: Put a page in the mode it belongs to. When a topic
   spans modes, place it in the most logical one and cross-link — do not duplicate.
4. **Tone**: *User docs* are practical and task-oriented; *admin docs* are technical
   and detail-oriented; *reference docs* are exhaustive and dry.
5. **Diagrams**: Use **D2** in `d2` code blocks within Markdown files.
6. **Cross-Linking**: Link from high-level guides to specific references to avoid
   duplication. Internal links use the absolute form `/scion/<slug>/`.

## 6. Verification

- **Sidebar**: Register new pages in `docs-site/astro.config.mjs` under the correct
  mode group.
- **Redirects**: When moving or renaming a page, add a redirect in the
  `redirects` map in `astro.config.mjs` so old URLs do not 404.
- **Link Check**: `starlight-links-validator` runs during `npm run build` (Node ≥ 22).
  Fix every reported broken link.
- **Formatting**: Standard Markdown. Starlight supports
  [Callouts](https://starlight.astro.build/guides/authoring-content/#asides) (Note,
  Tip, Caution, Danger).

**Note on D2**: The `d2` CLI may not be available in all environments. If you cannot
run `./check-d2.sh`, ensure your D2 syntax is correct per the
[D2 documentation](https://d2lang.com/tour/intro).
